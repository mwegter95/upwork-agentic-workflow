Let me answer your two required questions directly, because I know you skip applications that do not.

**Q1. Largest structured file processed in production, and the memory approach.** I have processed multi-hundred-MB structured exports in production where loading the full tree was never an option. The pattern I reach for is lxml.etree.iterparse with `events=('end',)` and `tag` set to the repeating element (here that is `Transaction` and `Invoice`). On each end event I process the element, then call `elem.clear()` to release its text and attributes, then prune the preceding siblings with `while elem.getprevious() is not None: del elem.getparent()[0]`. That keeps peak memory bounded to roughly one element subtree no matter how long the file runs, so a 500 MB SAF-T file with tens of thousands of transactions stays in the 80 to 120 MB range instead of the 600 to 800 MB a full `etree.parse()` would cost. The parsed records stream straight out to PostgreSQL (or to S3 for the raw artifact) in batches, so nothing accumulates. The memory-stability chart in the demo is a visualization of exactly this: the flat streaming line is `elem.clear()` plus ancestor pruning, the climbing line is the naive load. The real Milestone 2 deliverable proves it on an actual file.

**Q2. Tenant isolation in FastAPI + PostgreSQL, and at which layers.** I enforce it at four layers, and I test it.
1. The JWT (from your Supabase or Auth0 adapter) carries a `tenant_id` claim. A FastAPI dependency decodes it into a `TenantContext`, and every data route declares that dependency, so a route with no tenant context cannot touch tenant tables.
2. Every query runs through a tenant-scoped SQLAlchemy session that automatically appends `tenant_id == ctx.tenant_id` to selects, updates, and deletes, so no individual route can forget the filter. This is the primary layer and it is unit-testable per query.
3. PostgreSQL Row-Level Security is the defense-in-depth layer. Each table has a policy keyed on `current_setting('app.current_tenant')`, and the connection runs `SET LOCAL app.current_tenant = '<uuid>'` at the start of each transaction. `SET LOCAL` (not `SET SESSION`) matters under PgBouncer transaction pooling so context never leaks across reused connections. Even if the app filter were bypassed, the database refuses the other tenant's rows.
4. API-level isolation tests seed two tenants and assert that a request authenticated as tenant A cannot read, modify, or delete tenant B's records, including with manipulated tokens. Static analysis (ruff + bandit) flags raw SQL that could sidestep the ORM. Isolation is enforced and proven, not assumed.
